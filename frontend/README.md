# GoodVibes sidebar panel build

The GoodVibes Home Assistant integration ships a custom sidebar panel. Its
frontend assets are **authored in `src/`** and **built** into the integration's
served directory (`../custom_components/goodvibes/frontend/`) by a reproducible
esbuild step — they are not hand-edited bundles.

## Layout

| Path | Role |
| --- | --- |
| `src/goodvibes-home-panel.js` | Panel source (authored, checked in) |
| `src/goodvibes-icons.js` | Sidebar icon module source (authored, checked in) |
| `build.mjs` | esbuild build script |
| `../custom_components/goodvibes/frontend/*.js` | Built artifacts (generated, committed) |

## Build

```bash
cd frontend
npm ci
npm run build
```

This bundles each source module with esbuild (ES module output, not minified so
the served file stays reviewable), stamps it with the integration version read
from `manifest.json`, and writes the result into the served directory.

## Why the built artifact is committed

Home Assistant installs the integration by copying `custom_components/goodvibes/`
(via HACS or by hand) and never runs a build step, so the built artifacts must be
present in the repository. They are regenerated by `npm run build`, never edited
directly. To edit the panel, change the source in `src/` and rebuild.

## Reproducibility

* esbuild is pinned to an exact version in `package.json`, and `npm ci` installs
  from `package-lock.json`.
* `npm run check` (used in CI) rebuilds in memory and fails if the committed
  artifacts are stale, so a forgotten rebuild or a hand-edit to a served file is
  caught before merge. The release workflow also rebuilds and fails if the tree
  changes, so every published archive contains a fresh build.
