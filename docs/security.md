# Security and Credentials

GoodVibes uses three different credential roles for the Home Assistant surface. Keep them separate.

## Credential Roles

| Credential | Stored by | Used for |
| --- | --- | --- |
| Daemon bearer token | Home Assistant config entry | Authenticates Home Assistant requests to the GoodVibes daemon. |
| Home Assistant webhook secret | Daemon config and Home Assistant config entry | Authenticates calls to the integration webhook path. |
| Home Assistant long-lived access token | Daemon config | Lets the daemon call Home Assistant REST APIs for state, service, template, and event tools. |

The daemon bearer token is not the same thing as the Home Assistant webhook secret. The webhook secret is not a substitute for daemon API auth.

GoodVibes `goodvibes://...` secret references are daemon-side config values. They work for daemon-owned secrets such as `surfaces.homeassistant.accessToken` and `surfaces.homeassistant.webhookSecret`, but Home Assistant cannot use a daemon-local secret reference as the bearer value for its first HTTP request to the daemon. The integration needs a bearer token or session token value it can send in the `Authorization` header.

## Daemon Bearer Token

The current GoodVibes-native bearer-token path is the companion/operator token generated by the TUI/daemon pairing flow. GoodVibes stores the canonical token under the daemon home, usually `~/.goodvibes/daemon/operator-tokens.json`, with owner-only permissions.

Home Assistant stores the token value it receives because it must attach that value to daemon API requests. The config flow masks the token field.

Do not commit daemon URLs, bearer tokens, webhook secrets, or Home Assistant access tokens into this repository.

## Home Assistant Access Token

The Home Assistant long-lived access token belongs in daemon config:

```json
{
  "surfaces": {
    "homeassistant": {
      "instanceUrl": "http://homeassistant.local:8123",
      "accessToken": "goodvibes://..."
    }
  }
}
```

Use a Home Assistant user/token scope appropriate for daemon-side state, service, template, and event operations. Rotate the token if the daemon host, config files, logs, or backups may have exposed it.

## Webhook Secret

The webhook secret is shared between daemon config and the Home Assistant integration config. It protects the integration webhook path:

```http
POST /webhook/homeassistant
```

Use the same event type in both places when customizing the default `goodvibes_message` event type.

## Browser Boundary

The `GoodVibes Home` sidebar never receives the daemon token.

Browser actions use Home Assistant-authenticated entry points:

- Websocket command: `goodvibes/home_graph/call`
- Upload path: `POST /api/goodvibes/home-graph/upload`

Home Assistant forwards those requests to the daemon with the stored daemon bearer token. Keep this boundary intact when changing panel code.

## Upload Handling

Browser uploads use temporary files only for uploaded document/photo bytes. The integration forwards the file to the daemon for artifact storage and extraction, then unlinks the temporary file after the daemon ingest call finishes.

Do not base64 large PDFs, manuals, receipts, or photos into JSON. Use multipart uploads through the sidebar or daemon-local file paths where appropriate.

## Rotation

Rotate credentials when access may have been exposed or when moving a Home Assistant instance to a new host.

Recommended rotation order:

1. Generate or approve a new daemon companion/operator token.
2. Update the GoodVibes integration config entry with the new daemon bearer token.
3. Rotate the Home Assistant webhook secret in both daemon config and integration config.
4. Rotate the Home Assistant long-lived access token in daemon config.
5. Restart the daemon.
6. Restart Home Assistant after the daemon reports healthy.
7. Verify `GET /status`, `GET /api/homeassistant/health`, Assist, and Home Graph status.

Revoke old tokens after the new configuration is verified.

## Logs and Diagnostics

Avoid logging raw credential values. When collecting diagnostics, redact:

- Daemon bearer tokens
- Home Assistant long-lived access tokens
- Webhook secrets
- Full daemon URLs if they contain hostnames or internal network details you do not want to disclose
- Uploaded files that may contain receipts, manuals with serial numbers, photos, or private notes

Home Graph exports may include knowledge sources, facts, issue history, and links to Home Assistant objects. Treat exports as sensitive unless reviewed.

## Future Credential Shape

Preferred future shape:

1. Home Assistant starts a Home-Assistant-specific GoodVibes pairing flow from the config flow.
2. The daemon operator approves the pairing in GoodVibes.
3. The daemon issues a revocable, scoped Home Assistant integration credential.
4. Home Assistant stores only that scoped credential and can rotate or revoke it without using the shared operator token.
